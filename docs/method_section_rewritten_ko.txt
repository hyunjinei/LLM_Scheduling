제안 방법(Method) 초안

이 문서는 현재 코드 구현과 연구 기여점을 기준으로, 논문 본문에 들어갈 수 있는 방법론 초안을 정리한 것이다. 아래 내용은 기존 GPT 초안에서 부정확하거나 과장될 수 있는 표현을 정리한 뒤, 논문용 서술 형식으로 다시 작성한 버전이다.


1. 기존 초안에서 수정해야 하는 핵심 포인트

첫째, "step-level self-labeled dataset"이라는 표현은 현재 방법을 정확히 설명하지 못한다. 본 연구의 데이터는 모델이 자기 자신을 라벨링한 결과가 아니라, teacher schedule로부터 각 시점의 상태-행동 쌍을 추출해 만든 teacher-derived step supervision dataset에 가깝다. 따라서 방법론 본문에서는 self-labeled보다는 teacher-derived, teacher-guided, 또는 teacher-schedule-derived step dataset과 같은 표현을 사용하는 편이 정확하다.

둘째, masking 설명은 prefix-level 설명만으로 두지 않는 것이 좋다. 개념적으로는 decode-time feasible-action masking이라고 설명할 수 있지만, 현재 구현은 action code를 single special token으로 등록한 뒤 feasible action token id 집합만 남기는 direct token-set restriction에 가깝다. 따라서 논문에서는 일반 개념으로는 decode-time feasible-action masking이라 부르고, 현재 action representation에서는 이를 feasible token-set restriction으로 구현한다고 밝히는 것이 가장 정확하다.

셋째, 본 논문의 핵심 모델과 확장 모듈의 위계를 분리해 쓰는 것이 필요하다. 현재 프레임워크는 policy, reason, self-reflection, reinforcement learning까지 연결될 수 있지만, 주 정책은 어디까지나 policy-only masked scheduler이다. 따라서 방법론의 중심은 step-wise policy learning과 masked decoding에 두고, reason supervision, self-reflection prompting, reinforcement learning은 동일한 인터페이스 위에 얹을 수 있는 확장 모듈로 서술하는 것이 논리적으로 가장 깔끔하다.

넷째, 학습 objective 설명에 action-centric supervision을 명시해야 한다. 현재 구현은 단순한 chat continuation 학습이 아니라, assistant action token을 중심으로 supervision을 주는 방식으로 바뀌어 있다. 특히 action-only 모드에서는 선택된 action token만 학습하며, action-reason 모드에서도 첫 action token은 feasible action set 위 normalized cross-entropy로 학습되고, 나머지 reason token은 보조 supervision으로 다뤄진다. 이 점은 기존 GPT 초안에 빠져 있었으므로 방법론 본문에 포함되어야 한다.

다섯째, serial/dispatch를 모두 강하게 주장하는 문구는 현재는 조심하는 편이 좋다. 코드 수준에서는 둘 다 구현되어 있지만, 논문 메인 주장에서는 먼저 step-wise masked policy 자체를 중심으로 쓰고, 필요하면 environment instantiation이나 implementation note 수준에서 serial 또는 dispatch setting으로 적용 가능하다고 정리하는 편이 안전하다.


2. 논문용 방법론 초안

4. 제안 방법

4.1 전체 개요

본 연구는 Job Shop Scheduling Problem(JSSP)을 전체 스케줄 문자열을 한 번에 생성하는 문제로 다루지 않고, 현재 상태에서 다음 행동 하나를 선택하는 step-by-step action selection 문제로 재정의한다. 전체 프레임워크는 다음과 같은 흐름으로 구성된다. 먼저, teacher schedule이 포함된 JSSP 데이터를 step-level 상태-행동 데이터로 변환하여 학습용 dataset을 구축한다. 이후 각 step에서 현재 scheduling state와 feasible action set을 입력으로 받아 하나의 action code를 선택하도록 LLM policy를 LoRA 기반으로 학습한다. 추론 단계에서는 environment가 계산한 feasible action set 위에서만 decode-time masking을 적용하여 다음 행동을 선택하고, 선택된 행동은 즉시 환경에 반영되어 다음 상태를 형성한다. 마지막으로, 동일한 step-level 인터페이스를 유지한 채 self-reflection prompting 또는 reinforcement learning을 이용해 정책을 추가적으로 개선할 수 있다. 이때 본 논문의 핵심 정책은 policy-only masked scheduler이며, reason, reflection, RL은 그 위에 결합 가능한 확장 모듈로 다룬다.

4.2 Step-level environment-grounded reformulation

기존의 whole-schedule generation 방식에서는 모델이 문제 전체를 보고 "Job i Operation k" 형태의 전체 연산 시퀀스를 자유 형식 텍스트로 한 번에 생성한다. 그러나 이 방식은 긴 출력 시퀀스 전반에 걸쳐 선후행 제약, 기계 자원 제약, 중복 및 누락 방지, 형식 일관성을 모두 모델 내부에 동시에 유지해야 하므로, 문제 크기가 커질수록 오류 누적에 취약하다. 본 연구는 이러한 부담을 줄이기 위해, JSSP를 매 step에서 하나의 행동만 선택하는 정책 문제로 분해한다. 각 step t에서 환경은 현재 상태 s_t와 현재 feasible action set A(s_t)를 계산하고, 정책 π_θ는 이 집합 안에서 다음 행동 a_t를 선택한다. 이후 환경 전이는 s_(t+1)=T(s_t,a_t)로 정의된다. 이 구조에서는 전체 해의 일관성이 "문자열 전체를 한 번에 잘 생성하는 능력"이 아니라, "현재 상태에서 유효한 행동을 반복적으로 선택하는 능력"으로 전환된다. 따라서 feasibility와 상태 전이는 프롬프트 외부의 추상적 제약이 아니라, 환경과 행동 공간 정의에 직접 연결된 구성 요소가 된다.

4.3 Randomized step-local action code

본 연구는 절대 job index에 대한 출력 편향을 줄이기 위해 step-local randomized action code를 도입한다. 구체적으로, 각 step에서 현재 feasible action 집합 A(s_t)에 대해 무작위 일대일 대응 φ_t : A(s_t) -> C_t를 생성하고, C_t의 원소는 <A0410>, <A4507>와 같은 action code로 표현한다. 모델은 절대 job 번호를 직접 출력하는 대신 action code c_t ∈ C_t를 예측하며, 실제 환경에 적용할 때는 a_t = φ_t^{-1}(c_t)로 역매핑한다. 이때 동일한 job이라도 step이 달라지면 전혀 다른 code에 대응될 수 있으므로, 모델은 "Job 1", "Job 2"와 같은 고정 인덱스 패턴을 외우기보다 현재 상태와 후보 특성에 기반한 상대적 선택을 학습하게 된다. 이러한 설계는 특히 학습 시 다룬 문제 크기를 넘어서는 인스턴스에 대해, 절대 인덱스 기반 출력 편향을 줄이고 보다 상태 의존적인 정책 일반화를 유도하는 데 목적이 있다.

4.4 Decode-time feasible-action masking

본 연구의 핵심은 실행 가능성을 생성 후 검사하는 것이 아니라, 디코딩 단계에서 직접 통제하는 데 있다. 각 step에서 environment는 현재 상태 s_t로부터 feasible action code 집합 C_t를 계산한다. 디코딩 시에는 이 집합에 대응하는 토큰만 허용하고, 나머지 토큰은 모두 제거한다. 개념적으로 이는 decode-time feasible-action masking이며, 현재 구현에서는 action code가 single special token으로 등록되어 있으므로 feasible action code 집합 C_t를 token id 집합 V_t로 변환한 뒤 다음 토큰 분포를 V_t 위로 직접 제한하는 feasible token-set restriction으로 구현된다. 따라서 본 연구에서의 hard feasibility control은 "모든 현실 제약에 대한 절대적 보장"을 의미하는 것이 아니라, 환경이 모델링한 action space와 제약 범위 안에서 infeasible action이 생성 단계에서 차단된다는 뜻이다. 이는 validity를 sampling, repair, post-selection에 의존하는 기존 LLM 기반 접근과 가장 구별되는 지점이다.

4.5 상태 표현과 candidate transition features

본 연구의 정책 입력은 단순한 raw problem description이 아니라, 현재 상태와 후보 행동의 전이 효과를 함께 포함하는 구조화된 자연어 prompt이다. 먼저 static problem context에는 문제 크기와 총 operation 수와 같은 최소한의 problem-level 정보만 남긴다. 다음으로 dynamic state에는 현재 step index, current makespan, unfinished job 수, 남은 작업량, machine ready-time 통계, bottleneck machine과 remaining load 등이 포함된다. 마지막으로 각 feasible candidate에 대해 operation machine, processing time, estimated start/end time, projected makespan change, remaining work after action, machine load, post-route와 같은 candidate transition features를 함께 제공한다. 이러한 표현의 목적은 모델이 단순히 job identity를 이어 쓰는 것이 아니라, 각 후보를 선택했을 때 현재 상태가 어떻게 변하는지를 읽고 비교하도록 만드는 데 있다. 또한 candidate ordering bias를 줄이기 위해, visible candidate listing은 고정 휴리스틱 정렬 대신 무작위화된 순서로 제시된다.

4.6 Teacher-derived step supervision dataset

학습 데이터는 teacher schedule이 포함된 JSSP instance로부터 생성한다. 한 문제의 teacher solution을 전체 해 label 하나로만 사용하는 대신, 이를 J x M개의 step row로 분해하여 각 row마다 현재 상태, feasible action set, randomized action-code mapping, teacher action code, 그리고 선택적으로 reason target을 저장한다. 다시 말해, 하나의 whole-schedule label을 다수의 step-level supervision row로 변환함으로써 dense한 정책 학습 신호를 얻는다. 이 과정에서 각 step은 독립적인 텍스트 예시처럼 보이지만, 실제로는 동일한 teacher rollout의 상태 전이를 따라 생성된 환경 기반 데이터이다. 현재 프레임워크는 이를 policy, reason, both 세 종류의 supervision role로 구성할 수 있지만, 본 논문의 중심 설정은 policy-only이다.

4.7 Action-centric LoRA policy learning

본 연구의 supervised fine-tuning은 일반적인 chat continuation 학습이 아니라, 현재 상태에서 올바른 다음 action을 선택하는 정책 학습에 맞추어 구성된다. 각 step t에 대해 입력 x_t는 state summary와 candidate transition feature를 포함한 자연어 prompt이고, 목표 출력은 teacher action code c_t*이다. action-only 설정에서는 assistant span 전체에 loss를 주는 것이 아니라, 선택된 action token 하나만 supervision 대상으로 사용한다. 특히 현재 step의 feasible action token 집합을 V_t라 할 때, action token에 대한 학습 objective는 전체 vocabulary가 아니라 V_t 위에서 정규화된 확률을 사용하는 action-centric objective로 구성된다. 즉,

L_action = - log p_theta(c_t* | x_t, V_t)

와 같이 볼 수 있다. 이는 scheduling 문제를 자유 형식 텍스트 생성보다 "현재 feasible set 중 어떤 action이 더 적절한가"에 가까운 분류 문제로 다루기 위한 설계이다. LoRA는 이러한 정책 학습을 7B~8B급 LLM에서 실용적으로 수행하기 위한 parameter-efficient fine-tuning 수단으로 사용된다.

4.8 Reason supervision과 self-reflection prompting

본 논문의 주 정책은 policy-only masked scheduler이지만, 동일한 step-level 인터페이스 위에서 reason supervision과 self-reflection prompting을 확장적으로 결합할 수 있다. Reason supervision은 이미 선택된 action이 왜 타당한지, 그리고 강한 대안 후보를 왜 선택하지 않았는지를 자연어로 설명하도록 하여, 정책 선택의 해석 가능성과 보조 학습 신호를 제공한다. Self-reflection prompting은 1차 rollout이 완료된 후 스케줄 전체를 사후적으로 분석하여, final makespan에 불리하게 작용했을 가능성이 있는 critical decision step을 다시 검토하는 방식이다. 이때 reflection prompt에는 원래의 상태 설명, 이전에 선택된 action, 대안 후보, 그리고 완성된 스케줄을 분석해 얻은 병목 및 후속 경로 관련 정보가 함께 주어진다. 이를 통해 본 프레임워크는 전체 해를 새로 생성하지 않고도, 특정 step 이후의 의사결정을 다시 고려하여 rollout을 개선할 수 있다.

4.9 Reinforcement learning extension

Supervised policy가 학습된 이후에는 reinforcement learning을 통해 정책을 추가적으로 개선할 수 있다. RL 단계에서는 step-wise masked policy를 사용해 rollout을 수행하고, 최종 makespan에 기반한 보상으로 정책을 업데이트한다. 현재 구현은 REINFORCE, GRPO-style update, BOPO-style preference optimization을 포함할 수 있으며, 모든 경우에 rollout은 동일한 feasible-action masking과 environment transition을 전제로 한다. 따라서 RL은 free-form text generation을 최적화하는 것이 아니라, 이미 학습된 masked step-wise policy를 보다 낮은 makespan 방향으로 조정하는 단계로 해석할 수 있다. 본 논문에서는 이러한 RL 확장을 핵심 정책 구조 위에 추가될 수 있는 정책 개선 메커니즘으로 위치시킨다.

4.10 요약

정리하면, 본 연구의 방법론은 JSSP를 step-wise action selection 문제로 재정의하고, randomized step-local action code와 decode-time feasible-action masking을 결합하여 행동 단계에서 실행 가능성을 직접 통제하는 데 핵심이 있다. 이 위에 candidate transition feature 기반 상태 표현, teacher-derived step supervision, action-centric LoRA policy learning, 그리고 선택적으로 reason, self-reflection, reinforcement learning을 결합함으로써, 실행 가능성과 makespan 최소화를 동시에 겨냥하는 LLM 기반 스케줄링 프레임워크를 구성한다.
